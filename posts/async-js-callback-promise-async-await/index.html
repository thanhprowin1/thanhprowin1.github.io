Async JS: Hành trình từ Callback Hell đến Promise và Async/Await

Tại sao code JavaScript không chạy tuần tự? Bài viết này sẽ giải thích về bất đồng bộ và hành trình進化 của nó, từ 'địa ngục callback' rối rắm đến cú pháp thanh lịch của async/await.
October 10, 2025 · Hữu Khang

Giới thiệu: Tại sao JavaScript lại “làm việc nhà” khác người?#

Hãy tưởng tượng bạn gọi một ly cà phê. Trong thế giới đồng bộ (synchronous), bạn sẽ đứng chờ ở quầy cho đến khi có cà phê rồi mới làm việc khác. Điều này rất lãng phí thời gian.

JavaScript hoạt động theo kiểu bất đồng bộ (asynchronous), giống như một người phục vụ thông minh. Họ nhận order của bạn (bắt đầu một tác vụ), sau đó đi phục vụ các bàn khác trong khi ly cà phê của bạn đang được pha. Khi cà phê xong, họ sẽ mang ra cho bạn (hoàn thành tác vụ).

Cơ chế này giúp JavaScript không bị “treo” khi phải chờ các tác vụ tốn thời gian như gọi API, đọc file. Nhưng việc quản lý các tác vụ “sẽ hoàn thành trong tương lai” này đã trải qua một cuộc cách mạng.

1. Khởi đầu: Callback và “Địa ngục Callback Hell”#

Callback là một hàm được truyền vào một hàm khác như một tham số, và sẽ được gọi lại (call back) khi tác vụ hoàn thành.

Nghe thì đơn giản, nhưng khi bạn có nhiều tác vụ phụ thuộc lẫn nhau, code của bạn sẽ trông như một kim tự tháp lộn ngược, hay còn được gọi là “Callback Hell” (Địa ngục Callback).

Ví dụ về Callback Hell:#

Giả sử chúng ta cần làm 3 bước: lấy thông tin người dùng, sau đó lấy danh sách bài viết của người đó, rồi lấy tiếp bình luận của bài viết đầu tiên.

// Đây là code minh họa, không chạy được
getUser(1, (user) => {
  console.log('Tìm thấy user:', user.name);
  getPosts(user.id, (posts) => {
    console.log('Tìm thấy', posts.length, 'bài viết');
    getComments(posts[0].id, (comments) => {
      console.log('Tìm thấy', comments.length, 'bình luận');
      // Nếu có bước thứ 4, nó sẽ thụt vào sâu hơn nữa...
    });
  });
});

2. Promise: Lời hứa giải cứu#

Một Promise (lời hứa) là một đối tượng đại diện cho sự hoàn thành (hoặc thất bại) của một tác vụ bất đồng bộ. Nó có 3 trạng thái:

  • Pending: Lời hứa đang được thực hiện.
  • Fulfilled: Lời hứa đã hoàn thành thành công.
  • Rejected: Lời hứa đã thất bại.

Promise giúp chúng ta “làm phẳng” kim tự tháp callback bằng cách sử dụng chuỗi .then() để xử lý thành công và .catch() để bắt lỗi.

Ví dụ viết lại bằng Promise:#

// Đây là code minh họa, không chạy được
getUser(1)
  .then(user => {
    console.log('Tìm thấy user:', user.name);
    return getPosts(user.id); // Trả về một Promise khác
  })
  .then(posts => {
    console.log('Tìm thấy', posts.length, 'bài viết');
    return getComments(posts[0].id); // Trả về một Promise khác
  })
  .then(comments => {
    console.log('Tìm thấy', comments.length, 'bình luận');
  })
  .catch(error => {
    // Bắt tất cả lỗi ở một nơi duy nhất!
    console.error('Đã có lỗi xảy ra:', error);
  });

3. Async/Await: Cú pháp thanh lịch nhất#

Async/Await là một lớp “đường cú pháp” (syntactic sugar) được xây dựng dựa trên Promise, giúp code bất đồng bộ trông giống hệt như code đồng bộ.

  • async: Đặt từ khóa này trước một hàm để biến nó thành một hàm bất đồng bộ (luôn trả về một Promise).
  • await: Chỉ có thể dùng bên trong một hàm async. Nó sẽ “tạm dừng” việc thực thi của hàm cho đến khi Promise được giải quyết và trả về kết quả.
  • Xử lý lỗi: Chúng ta dùng khối try...catch quen thuộc.

Ví dụ viết lại bằng Async/Await:#

// Đây là code minh họa, không chạy được
async function fetchUserData() {
  try {
    const user = await getUser(1);
    console.log('Tìm thấy user:', user.name);

    const posts = await getPosts(user.id);
    console.log('Tìm thấy', posts.length, 'bài viết');

    const comments = await getComments(posts[0].id);
    console.log('Tìm thấy', comments.length, 'bình luận');
  } catch (error) {
    console.error('Đã có lỗi xảy ra:', error);
  }
}

fetchUserData();

Kết luận#

Hành trình từ Callback Hell đến Promise và cuối cùng là Async/Await cho thấy sự phát triển của JavaScript trong việc xử lý bất đồng bộ ngày một tốt hơn. Ngày nay, Async/Await được xem là tiêu chuẩn vàng vì nó cung cấp cách viết code dễ đọc, dễ hiểu và dễ bảo trì nhất cho các tác vụ bất đồng bộ.
